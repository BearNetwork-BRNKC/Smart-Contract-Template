# BNES-ERC20: Physics-Informed Token Template for BearNetworkChain (BNES)

## Overview

`BNES-ERC20` is the official ERC20 deployment template recommended by **BearNetworkChain (BNES)**. This template integrates **physics engine alignment** with **post-quantum cryptography (PQC)** verification, ensuring that all token operations are anchored to BNES's underlying physical field while maintaining quantum-resistant security guarantees.

---

## Key Features

### 1. Physics Engine Alignment (18 Decimal Precision)

- **Hard-bound precision**: All token transfers, mints, and burns are projected to BNES's physics engine via the `IBNESPhysicsCore` interface.
- **Zero drift**: The contract guarantees that the EVM state and the physics field state remain perfectly synchronized, preventing RF-1 invariant anomalies.
- **Immutable precision**: The `decimals()` function is never overridden; the contract always uses 18 decimals as required by BNES.

### 2. Post-Quantum Cryptography (PQC) Verification

- **Dilithium-v3 signatures**: All privileged operations (mint, burn, blacklist, bridge) are protected by `onlyQuantumSafe`, which verifies that the transaction origin possesses a valid post-quantum signature.
- **Quantum envelope transactions**: BNES nodes automatically wrap user transactions into `QuantumEnvelopeTx`, making PQC verification transparent to end users.

### 3. Cross-Chain ZK Bridge Integration

- **Halo2-KZG proofs**: The `bridgeMint` function requires a zero-knowledge proof generated by the Halo2 circuit, ensuring that cross-chain assets maintain 18-decimal precision alignment.
- **Replay protection**: Each bridged transaction is marked as processed to prevent replay attacks.

### 4. Production-Ready Business Logic

- **Fee-on-transfer**: Correctly implemented with dual physical mapping to conserve total flux.
- **Rate limiting**: Prevents MEV front-running and bot abuse.
- **Whitelist/blacklist**: Configurable access control for deployments.
- **DAO governance**: Optional multisig minting via `onlyGovernance` modifier.

### 5. Security Hardening

- **Reentrancy immune**: Follows Checks-Effects-Interactions pattern.
- **Flash loan safe**: BNES's physics engine detects any precision drift caused by flash loan arbitrage and forces revert.
- **Integer overflow/underflow safe**: Uses Solidity `^0.8.27` with built-in SafeMath.
- **Privilege escalation protected**: All privileged functions require PQC verification.

## Usage

### Minting Tokens

```solidity
uint256 MAX_SUPPLY = 1_000_000_000 * 1e18; // 1 billion cap

function mint(address to, uint256 amount) external onlyOwner onlyQuantumSafe {
    require(totalSupply() + amount <= MAX_SUPPLY, "Exceeds max supply");
    _mint(to, amount);
}
```

### Adding Liquidity to DEX

```solidity
// Approve DEX router
await myToken.approve(BNS_DEX_ROUTER_ADDRESS, ethers.MaxUint256);

// Add liquidity (tokenAmount must be 18-decimal wei)
const tokenAmount = ethers.parseUnits('100000', 18);
const bnsCoinAmount = ethers.parseEther('10');

await dexRouter.addLiquidityETH(
    myToken.address,
    tokenAmount,
    tokenAmount * 95n / 100n,  // 5% slippage
    bnsCoinAmount * 95n / 100n,
    ownerAddress,
    Date.now() + 3600          // 1-hour deadline
);
```

### Bridging from Another Chain

```solidity
function bridgeMint(
    address to,
    uint256 amount,
    bytes calldata zkWitness,
    bytes32 stateRoot,
    bytes32 srcTxHash
) external onlyQuantumSafe {
    require(!bridgePaused, "Bridge is paused");
    require(msg.sender == tokenBridge, "Only bridge relayer");
    require(!processedBridgeTx[srcTxHash], "Tx already processed");

    if (!IBNESPhysicsCore(BNES_CORE).verifyPhysicalWitness(zkWitness, stateRoot)) {
        revert InvalidZKProof();
    }

    processedBridgeTx[srcTxHash] = true;
    _mint(to, amount);
    emit BridgeMinted(to, amount, srcTxHash);
}
```

## Security Considerations

### Do Not Deploy on Non-BNES Chains

This contract relies on the custom BNES precompile at `0x0000000000000000000000000000000000000F15`. Deploying it on Ethereum, BSC, Polygon, or any other EVM chain will cause all transfers to revert permanently, turning the token into a "zombie token."

### Do Not Modify Immutable Rules

- **Never** change `BNES_CORE` to a variable or make it updatable.
- **Never** use `msg.sender` instead of `tx.origin` in `onlyQuantumSafe`.
- **Never** override `decimals()` to any value other than 18.
- **Never** call `projectFlux` before `super._update` or in partial branches.

### Extending the Contract

If you add custom privileged functions (e.g., `onlyOwner`), **always** include the `onlyQuantumSafe` modifier. Missing even one instance will break the security loop and expose the contract to quantum attacks.
